5. AUTHENTICATION AND AUTHORIZATION CONTRACT
==================================================
Auth API
- POST /api/auth/register
  - customer payload: first_name, last_name, email, phone, password, role=customer
  - provider payload: first_name, last_name, email, phone, password, role=provider, nid
- POST /api/auth/login
  - returns token and user {id, first_name, last_name, role}

Auth middleware behavior
- Bearer token required on protected routes.
- JWT payload includes id and role.

Current protected routes
- GET /api/bookings/provider
- GET /api/bookings/provider/:providerId
- POST/PUT /api/bookings/verify-handshake
- POST /api/emergency (middleware currently required)
- GET /api/admin/providers
- POST /api/admin/providers/:profileId/verify

Required upgrades
- Enforce ownership checks:
  - customer can only create bookings for self unless admin.
  - provider can only verify handshake for own bookings.
- Add refresh-token strategy or hardened session handling.
- Add brute-force protection and login rate limiting.
